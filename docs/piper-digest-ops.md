# Piper morning digest — operational setup

Phase 3 of Piper's EVOLUTION. The digest is generated by the **dai** service (the
Piper agent runtime lives here, not in bmad) and posted to `#piper` as Piper.

```
  Droplet systemd timer            dai service (already running on the droplet)
  Mon-Fri 09:00 ET                 ┌──────────────────────────────────────────┐
        │                          │  Hono API (src/api/server.ts, API_PORT)   │
        │  curl localhost          │   POST /api/cron/piper-digest              │
        └─────────────────────────▶│     → runPiperDigest()                     │
           Bearer ${CRON_SECRET}   │       → runAgent('piper', digest prompt)   │
                                   │       → getDedicatedBotClient('piper')     │
                                   │          .chat.postMessage(#piper)         │
                                   └──────────────────────────────────────────┘
```

Why dai, not a bmad Vercel endpoint: the agent, its Notion token, the Supabase
mirror tools and Piper's Slack bot client all live in dai. `slackTools.postMessage`
uses the **main** dai bot token, so the post is routed through Piper's dedicated
bot client instead — that's why posting happens in `runPiperDigest`, not via the
agent's `post_message` tool.

## Prerequisites (one-time, manual — see the Slack-registration checklist)

In `~/dev/dai/.env` on the droplet (and locally for testing):

```
PIPER_BOT_TOKEN=xoxb-...        # from the registered Piper Slack app
PIPER_APP_TOKEN=xapp-...        # from the registered Piper Slack app
PIPER_CHANNEL_ID=C0XXXXXXX      # the #piper channel ID (Slack → channel → "Copy channel ID")
CRON_SECRET=<long-random-string>  # must match the systemd EnvironmentFile below
```

After setting these, restart the dai service so it picks them up and boots the
Piper dedicated bot:

```bash
sudo systemctl restart dai      # adjust to the actual dai unit name on the droplet
```

`API_PORT` and `STUDIO_API_KEY` are already set, so the Hono server is already up
(it receives Fireflies webhooks). Confirm the cron endpoint is reachable:

```bash
# dry-run: generates the digest, returns it in the response, posts nothing
curl -fsS -X POST \
  -H "Authorization: Bearer ${CRON_SECRET}" \
  "http://127.0.0.1:${API_PORT:-3001}/api/cron/piper-digest?dry_run=1" | jq .

# real post to #piper
curl -fsS -X POST \
  -H "Authorization: Bearer ${CRON_SECRET}" \
  "http://127.0.0.1:${API_PORT:-3001}/api/cron/piper-digest" | jq .
```

The agent run takes ~60-150s (Notion pagination), so give curl `--max-time 300`.

## systemd timer

`/etc/piper-digest.env` (root-owned, `chmod 600`):

```ini
CRON_SECRET=<same value as in dai/.env>
API_PORT=3001
```

`/etc/systemd/system/piper-digest.service`:

```ini
[Unit]
Description=Piper morning digest (generate + post to #piper)
After=network-online.target

[Service]
Type=oneshot
EnvironmentFile=/etc/piper-digest.env
ExecStart=/usr/bin/curl -fsS --max-time 300 \
  -X POST \
  -H "Authorization: Bearer ${CRON_SECRET}" \
  http://127.0.0.1:${API_PORT}/api/cron/piper-digest
```

`/etc/systemd/system/piper-digest.timer`:

```ini
[Unit]
Description=Run Piper morning digest Mon-Fri 09:00 ET

[Timer]
# America/New_York handles EST/EDT automatically (systemd >= v252).
OnCalendar=Mon..Fri 09:00 America/New_York
Persistent=false

[Install]
WantedBy=timers.target
```

> If the droplet's systemd is older than v252 and rejects the timezone suffix,
> drop `America/New_York` and use `OnCalendar=Mon..Fri 13:00 UTC` — but note that
> hardcodes EDT and will be off by an hour during EST (Nov-Mar). Prefer the TZ
> suffix if supported (`systemctl --version`).

Enable:

```bash
sudo cp piper-digest.env /etc/piper-digest.env && sudo chmod 600 /etc/piper-digest.env
sudo systemctl daemon-reload
sudo systemctl enable --now piper-digest.timer
systemctl list-timers piper-digest.timer        # confirm next run
sudo systemctl start piper-digest.service        # fire once manually to test
journalctl -u piper-digest.service -n 50 --no-pager
```

## Notes

- `Persistent=false` on purpose: a missed 09:00 (droplet down) should not fire a
  stale digest at an odd hour. Next weekday is fine.
- The endpoint is idempotent only in the sense that re-running posts a fresh
  digest — there's no dedup. Don't double-schedule.
- Logs: `journalctl -u dai` shows the agent run + post; the route logs
  `piper-digest cron failed` on error with the message.
